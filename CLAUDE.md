## Project Overview
Vantage is a member-facing learning platform built for MEGA (Middle East Growth Academy).
Live URL: vantage.mega-mentorship.com
Stack: Vanilla React 18 (UMD/CDN), Babel Standalone, Supabase JS v2, no bundler.
All JSX files are loaded via `<script type="text/babel" src="...">` in index.html.
Hosted on Vercel (vercel.json: `framework: null` + rewrite `/legal → /legal.html`). Git repo is initialized.

## Static Pages
- `legal.html` — standalone Privacy Policy + Terms of Use + **Parental Consent** page served at `vantage.mega-mentorship.com/legal` (Vercel rewrite). Self-contained HTML with inline styles + Vantage blue mark in the header (inline SVG, cropped viewBox `190 170 645 645`). THREE tabbed sections (Privacy / Terms / Parental Consent). Updated whenever the legal copy changes; the in-app `LegalConsentModal` links here. Age policy (Privacy §5, Terms §2/§3): 16+ self-serve, 16–17 parental consent, 13–15 institutional exception (manual review), under-13 hard block. The **Parental Consent** tab ends in a token-routed consent form (embeds `SUPABASE_URL` + anon key): reading `?consent_for=<token>#consent` auto-lands on the tab and calls `minor-consent-resolve` to pre-fill the member's name/DOB, then `minor-consent-submit` on submit. See the Parental Consent / Age Gate section.

## File Architecture
Script load order in index.html is strict — do not reorder:
1. supabase.jsx — Supabase client init; exposes `window._supabase`. MUST load first.
2. data.jsx — Shared constants, SUBJECTS map, mock MEMBER object, Icon component (Tabler-style inline SVGs), utilities.
3. shell.jsx — Sidebar, Topbar, SidebarWithWins wrapper, Avatar component.
4. calendar.jsx, tour.jsx, modals.jsx, eisenhower.jsx, milestone.jsx — Supporting components.
5. dashboard.jsx, tasks.jsx, sessions.jsx, wins.jsx, resources.jsx, chat.jsx, profile.jsx, corporate-radar.jsx — Member views.
6. admin.jsx, adminplus.jsx, adminchat.jsx, analytics.jsx — Admin views.
7. Inline `<script type="text/babel">` in index.html — Login screen + root App component.

There are no imports between files. All components are globals on the window scope via Babel.
- `supabase/functions/` — source mirror of the deployed Edge Functions (not served to the browser; for version control / redeploys).
- `adminchat.jsx` is a thin alias (`AdminChatScreen = AdminChat`). The admin "Chat" page chains `admin-messaging` → `BulkMessaging` → `AdminChatScreen` → `AdminChat`, which now simply renders the member `ChatScreen`. The separate admin Channels page (`admin-chat`) was removed.
- Cross-file helpers live on `window`: `fetchSessions/fetchCalSessions/fetchListSessions/mapToCalSession/mapToListSession/AddToCalendar/fmtH` (calendar.jsx); `fetchFocusSessions/focusYmd/focusWeekStart/fmtMins/focusTimerRemaining/useNowTick` (dashboard.jsx); `mapTaskRow` (tasks.jsx); `getActiveUserId`/`_viewingAsId` (index.html — see View As below).

### View As (admin impersonation, read-only)
- Triggered from the Members table OR the member detail page (`AdminMemberDetail`). Sets `viewingAs = { id, name, initials, color, plan }`; `isRealAdmin` stays true so the real admin's capabilities are preserved.
- A persistent banner (`RemoteBanner`, rendered inside `.main-inner` above the Topbar so it appears on every route) reads exactly **"You are viewing as [Name]"** with a clearly visible **Exit** button (returns to `admin-members`).
- **Data plumbing:** App holds `viewProfile` (the impersonated member's full profile). When `viewingAs` toggles, App fetches that profile and calls `loadUserData(viewingAs.id)` which refetches tasks/goals/upcoming sessions for the member. The `member` object (used by every screen) derives from `viewProfile || liveProfile` for the visible fields (plan, xp, streak, role, interests, etc.) while `member.isAdmin` stays bound to `liveProfile.role==='admin'` (real capability).
- **Effective user id** is mirrored on `window._viewingAsId` and exposed via `window.getActiveUserId()` (returns `_viewingAsId` if set, else `auth.getUser().id`). Child fetchers that scope reads by user id use this helper: `fetchFocusSessions` (dashboard), `DashStats` attendance, `ProfileScreen` badge effect, `RoadmapScreen`, `fetchResourceViewCount`, `fetchCommunityPostCount`, and `calendar.fetchSessions` (which now applies `attendee_id.eq.<uid>,attendee_id.is.null` explicitly — needed because admin's `is_admin()` SELECT override would otherwise return ALL sessions).
- `.main-inner` is wrapped in a `React.Fragment` with `key={effectiveUserId}` so every screen unmounts/remounts on View As toggle — this forces all `useEffect([])` data loaders (RoadmapScreen, ProfileScreen, etc.) to re-run with the impersonated id without per-file dep changes.
- **Writes remain on the real admin's auth.uid()** by design (read-only): if the admin clicks a mutator in View As mode, the write targets the member's row and RLS rejects it. Buttons are not visually disabled (acceptable trade-off; if needed, gate UI mutators on `viewingAs`).

## Supabase Integration
URL and anon key are in supabase.jsx. Client is `window._supabase`.

Tables wired to live data:
- `profiles` — member auth, role (admin/member), membership_tier ('breakthrough'/'foundations'/'management'), account_type, trial status, member_status, joined_at, `google_connected`, `google_email`. Personal-info cols (editable from Profile, drive the Profile Complete badge): `job_title`, `employment_status`, `field`, `focus_area`, `interests` (text[]). `notification_preferences` (jsonb, with defaults) backs the Profile → Settings notification toggles. XP cols: `xp_total` (int, default 0 — cumulative XP, additive only; tier is DERIVED from it, never stored), `streak_days` + `last_streak_date` (weekday-streak engine state, also powers the dashboard Streak widget). **`onboarded_at` (timestamptz, nullable)** — server-side first-login gate: App shows the onboarding wizard only when this is null; completion writes `now()` to this column. Existing accounts were backfilled to `joined_at` so they don't re-trigger onboarding. **`legal_accepted_at` (timestamptz, nullable)** — server-side legal consent gate: App renders `LegalConsentModal` (modals.jsx) blocking the onboarding wizard, reflection modal, intention modal, and tour bubble whenever this is null. Accept writes `now()` to the column. The column was added with no backfill so EVERY existing account is NULL and will be prompted once on next login. The modal links to `/legal` (Privacy Policy + Terms of Use). (Note: the brief's `xp_regeneration_count` was intentionally SKIPPED — regeneration is already tracked per-task in `tasks.regen_count`.) **Age-gate cols:** `date_of_birth` (date, nullable — captured in the onboarding wizard), `age_gate_state` (text, CHECK null|`cleared`|`awaiting_consent`|`manual_review`|`blocked` — drives the member lock screen + admin members-list flag; **every existing profile was backfilled to `cleared`** so the gate can never lock an existing member), `partner_source` (text, nullable — institutional source org, e.g. TKS/DIFC Academy). See the Parental Consent / Age Gate section.
- `tasks` — member tasks. `roadmap_step_id` is now NULLABLE (standalone tasks allowed). Added columns: `goal_id` (direct goal link), `subject`, `priority`, `regen_count` (AI subtask regenerations used, max 3). `points`/`impact` are still client-side defaults (no columns) and `impact` is no longer rendered anywhere; completion uses the real `is_completed` column. Editable from the Tasks detail panel via `EditTaskModal` (title/notes/subject/priority/due_date/goal_id).
- `subtasks` — smallest unit of work, lives exclusively under a task. Cols: `task_id` (FK → tasks, ON DELETE CASCADE), `user_id`, `text`, `is_completed`, `source` (`ai`/`manual`), `order_index`, `created_at`. Own-row RLS (select/insert/update/delete) + `is_admin()` override on select/delete. Loaded lazily per task in `TaskDetailPanel`; AI ones come from the `generate-subtasks` Edge Function, manual ones from the inline add input. AI tag = `--sapphire`, Manual tag = `--coral`.
- `goals` — member goals (title, description, target_date, status).
- `roadmap_steps` — middle layer between goals and tasks (legacy GoalCard link path via `goal_id` OR `roadmap_step.goal_id`; also the phase layer for the Roadmap feature — `phase_label`, `reflection_prompt`, `admin_notes`, `order_index`, `status`, week range stored as prefix of `description`).
- `sessions` — 1:1s + town halls. Admin creates; members read own 1:1s (`attendee_id = auth.uid()`) + all town halls (`attendee_id` null) via RLS; admins have `is_admin()` SELECT override (View As filters explicitly by attendee_id). Cols: type, title, session_date, start_time, end_time, mentor_name, meeting_link, recurrence, attendee_id, created_by, `google_event_id`, **`agenda`** (text, nullable — plain-text bullets, one per line; appended to the Google Calendar invite description by `google-calendar-event` and to .ics/Google/Outlook exports via `sessionDescription()`).
- `wins` — community Wins board (load + post). Author info denormalized onto rows (`author_name`, `author_role`, `subject`) since profiles RLS is own-row-only. `is_public` gates the feed.
- `focus_sessions` — focus-timer "Save" writes here from BOTH the dashboard pomodoro and the Tasks-page panel timer (duration_minutes = actual elapsed, `selected_minutes` = chosen preset, label, subject, linked_id, linked_kind, started_at). For task-panel sessions `linked_kind='task'` / `linked_id=task.id`; the panel's cumulative "X h Ym focused on this task" line sums duration_minutes for that id. Read by the Focus metrics tab + dashboard "Focus this week" widgets.
- `win_reactions` — persisted reactions on wins (win_id, user_id, emoji; unique per trio). SELECT open to authenticated (counts + Win Of The Week computable); insert/delete own. Powers Wins reactions + the Win Of The Week badge.
- `resources` — the Resources Library content. Cols: id, title, description, `content_type` (`video`/`pdf`/`template`/`article`), `folder` (`foundations`/`breakthrough`/`mega_management`), subject_area, `access_tier` (same enum as folder), url, thumbnail_url, duration_minutes (videos only), published (default true), created_at. Added cols: `storage_path` (text, nullable — path in the private `resource-files` Storage bucket for directly-uploaded files; `url` is now NULLABLE with a `resources_source_present` CHECK requiring url OR storage_path). RLS: `resources_admin_all` (admins read/write everything via `is_admin()`) + `resources_member_read` — rewritten for bespoke assignments: a member SELECTs a row when `published=true` AND (`resource_assigned_to_me(id)` OR (`NOT resource_has_assignees(id)` AND tier rule [`access_tier in ('foundations','breakthrough')` OR `access_tier='mega_management'` AND `membership_tier='management'`])). I.e. a resource with ANY assignment is private to its assignees and ignores tier/folder; otherwise the original tier gate applies. `resource_assigned_to_me`/`resource_has_assignees` are SECURITY DEFINER (anon EXECUTE revoked) so the existence checks bypass `resource_assignments` RLS — without that, a member can't "see" that a resource is assigned to someone else and the bespoke resource leaks back into tier visibility (caught + fixed during build). Breakthrough-tier rows ARE returned to Foundations members — the lock is a client-side UX gate per the brief. Powers admin Resource Manager + member Library. The legacy `RESOURCES` mock in data.jsx is unused.
- `resource_assignments` — bespoke per-member resource assignments. Cols: id (uuid PK), `resource_id` (FK resources, CASCADE), `member_id` (FK profiles, CASCADE), created_at. UNIQUE(resource_id, member_id). RLS: `resource_assignments_admin_all` (`is_admin()`) + `resource_assignments_member_read` (own rows, `member_id = auth.uid()`). Written by AdminResources on save (diff sync); read by the member Library's "Shared with you" folder.
- Storage bucket `resource-files` (PRIVATE) — holds directly-uploaded resource files. Policies on `storage.objects`: `resource_files_admin_write` (ALL, `is_admin()`) + `resource_files_auth_read` (SELECT, any authenticated). Members never get a public URL — files are fetched via short-lived signed URLs (`createSignedUrl`, 1h); discovery is gated by the `resources` row RLS + unguessable UUID paths.
- `minor_consents` — parental-consent records for under-18 members. Cols: id (uuid PK), `member_id` (FK profiles, CASCADE), parent_name, parent_relationship, parent_email, parent_phone, partner_source, three SEPARATE consent booleans (`consent_data_processing`/`consent_program_participation`/`consent_terms_acknowledged`), signature_name, `consent_ip` + `consent_timestamp` (captured server-side, never trusted from the client), `consent_token` (text UNIQUE — opaque random token; member id is never in the URL), `token_expires_at`, `status` (CHECK `pending`/`submitted`), created_at. parent_* are nullable (a `pending` row is minted when the admin sends the request, filled when the parent submits). RLS: **no client writes** — only the service-role Edge Functions write here; `minor_consents_admin_read` (`is_admin()` SELECT) exposes records to the admin member-detail panel. Written/read by the three `minor-consent-*` Edge Functions; the admin detail panel reads the latest row per member.
- `resource_views` — one row per distinct resource a user opened (user_id, resource_id text; unique). Own-row RLS. Powers the Resources badge category. Logged via `window.logResourceView`; counted via `window.fetchResourceViewCount`.
- `channels` — chat channels (`name`, `description`, `created_by`, `is_default`, `is_dm`, `created_at`). One seeded `is_default` channel ("MEGA Mentorship") everyone shares. ## Parental Consent / Age Gate (live; backend deployed)
Age-gates minor members and captures verified parental consent. General access floor is **16**.
- **Tables/cols:** `profiles.date_of_birth`, `profiles.age_gate_state` (null|`cleared`|`awaiting_consent`|`manual_review`|`blocked`), `profiles.partner_source`; `minor_consents` table (service-role-write only + admin SELECT). All existing profiles backfilled to `cleared` — **the gate never affects an existing member**.
- **DOB capture:** OnboardingWizard Profile step (modals.jsx, required field) — the universal first-login chokepoint for BOTH self-serve and admin-invited members. `onComplete({dateOfBirth})` (index.html) computes the band via `window.ageFromDob` (data.jsx) and writes the gate, **never downgrading an already-`cleared` account**.
- **Bands:** 18+ → `cleared` + activate; 16–17 → `awaiting_consent` (lock screen, admin sends consent email); 13–15 → `manual_review` (lock screen, admin Approve/Keep-blocked); <13 → `blocked` + sign out.
- **Enforcement:** `loadProfile` only auto-activates `pending`→`active` when `age_gate_state==='cleared'`; a full-screen lock renders ahead of the app for the three gated states, keyed on the REAL `liveProfile` (admins + View-As exempt).
- **Consent flow:** admin opens `AdminMemberDetail` → "Age & parental consent" panel → enters parent email → `minor-consent-request` mints an opaque token (stored in `minor_consents` against `member_id`; never in the URL) + emails the guardian a `legal.html?consent_for=<token>#consent` link. Parent's page calls `minor-consent-resolve` (pre-fill) then `minor-consent-submit` (validates, captures server-side IP+timestamp, flips member to `active`+`cleared`). The email is **admin-triggered, not automatic** (no signup webhook exists; deliberate human-in-the-loop). `MembersTable` shows Review/Consent/Under-13 chips so the queue is visible.
- **Design deviations (approved):** dedicated `age_gate_state` column (not overloading `member_status`); DOB at onboarding (not signup), so under-13 is blocked right after first-login DOB entry; admin-sent consent email; `minor_consents` parent_* cols nullable (pending row precedes the parent's fill).

## Key Patterns and Conventions
- State lives in the root App component in index.html; passed down as props.
- Route persistence: `localStorage.getItem('vantage-route')`.
- **Onboarding gate is server-side** via `profiles.onboarded_at` — App sets `showOnboarding=true` only when the loaded profile has `onboarded_at IS NULL`. Tour/intention/reflection state still uses localStorage (per-browser is fine for those).
- **Legal consent gate is server-side** via `profiles.legal_accepted_at` — App sets `showLegalConsent=true` whenever it's null and renders `LegalConsentModal` ahead of (and blocking) every other modal/screen. Accept writes `now()`. Same pattern as the onboarding gate; the two run in sequence (legal first, then onboarding).
- **Age gate (parental consent) is server-side** via `profiles.age_gate_state` — see the Parental Consent / Age Gate section. DOB is captured in the OnboardingWizard (the universal first-login chokepoint covering both self-serve signup AND admin-invited members); `onComplete` computes the gate via `window.ageFromDob` (data.jsx) and writes `date_of_birth`/`age_gate_state`/`member_status`, **but never downgrades an already-`cleared` account** (existing-member safety). `loadProfile` only auto-activates `pending`→`active` when `age_gate_state==='cleared'` (brand-new accounts defer until onboarding resolves the gate). A full-screen age-gate lock (awaiting_consent / manual_review / blocked) renders ahead of the app, keyed on the REAL `liveProfile` so admins + View-As are never locked; under-13 is signed out. The consent email is **admin-triggered** (`AdminMemberDetail` "Send consent request"), not automatic — the Members-list flag (`MembersTable` Review/Consent chips) is the visible queue. 13–15 are resolved by the admin's Approve/Keep-blocked buttons.
- Tasks and goals are lifted to App state and passed to both DashboardScreen and TasksScreen.
- Global rewards animation uses `rewards` state array in App (point-pop XP display).
- `SidebarWithWins` wraps `Sidebar` from shell.jsx — use this wrapper, not Sidebar directly.
- Do not use ES module imports. All shared functions must be globally accessible.
- Class names in styles.css are not namespaced — check for collisions before adding new ones.
- **View As effective user id**: any child screen that needs to scope reads to the current user must use `await window.getActiveUserId()` (returns `_viewingAsId` if set, else the auth user id), NOT `auth.getUser().user.id`. The latter always returns the real admin during View As. Writes should keep using `auth.getUser()` — they intentionally fail under RLS when impersonating (read-only enforcement).

### Shared Focus Timer
- There is ONE active focus timer for the whole member app. Its state (`focusTimer`) + the `focus` API (`start/pause/resume/save/reset`) + a `focusTick` counter live in the root App (index.html) and are passed to BOTH `DashboardScreen` and `TasksScreen` as `focus` / `focusTick`. The dashboard pomodoro and the Tasks-panel timer are just views over it, so a session is mirrored across pages and survives collapsing a row / navigating between pages. Starting a new timer replaces any running one (single active session).
- Wall-clock based: the timer object stores `endsAt` (ms) when running and a frozen `remainingSec` when paused; `window.focusTimerRemaining(t)` derives the live remaining (accurate across background-tab throttling). It does NOT survive a full page refresh (no localStorage persistence — that was deliberately out of scope).
- Per-second re-renders are localised via `window.useNowTick(active)` (a 1s force-update used only by the components actually showing a countdown), so App does not re-render every second. Collapsed task rows show a live `⏱ MM:SS` pill when their timer is running.
- `focus.save()` writes one `focus_sessions` row (elapsed minutes + selected_minutes + linked task) and bumps `focusTick`; `DashboardScreen` reloads its focus rows on `focusTick` and `TaskDetailPanel` re-fetches its cumulative total on `focusTick`.
