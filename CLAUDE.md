# Vantage by MEGA ÔÇö Project Context

## Project Overview
Vantage is a member-facing learning platform built for MEGA (Middle East Growth Academy).
Live URL: vantage.mega-mentorship.com
Stack: Vanilla React 18 (UMD/CDN), Babel Standalone, Supabase JS v2, no bundler.
All JSX files are loaded via `<script type="text/babel" src="...">` in index.html.
Hosted on Vercel (vercel.json: `framework: null` + rewrite `/legal Ôćĺ /legal.html`). Git repo is initialized.

## Static Pages
- `legal.html` ÔÇö standalone Privacy Policy + Terms of Use page served at `vantage.mega-mentorship.com/legal` (Vercel rewrite). Self-contained HTML with inline styles + Vantage blue mark in the header (inline SVG, cropped viewBox `190 170 645 645`). Two tabbed sections (Privacy / Terms). Updated whenever the legal copy changes; the in-app `LegalConsentModal` links here.

## File Architecture
Script load order in index.html is strict ÔÇö do not reorder:
1. supabase.jsx ÔÇö Supabase client init; exposes `window._supabase`. MUST load first.
2. data.jsx ÔÇö Shared constants, SUBJECTS map, mock MEMBER object, Icon component, utilities.
3. shell.jsx ÔÇö Sidebar, Topbar, SidebarWithWins wrapper, Avatar component.
4. calendar.jsx, tour.jsx, modals.jsx, eisenhower.jsx, milestone.jsx ÔÇö Supporting components.
5. dashboard.jsx, tasks.jsx, sessions.jsx, wins.jsx, resources.jsx, chat.jsx, profile.jsx, corporate-radar.jsx ÔÇö Member views.
6. admin.jsx, adminplus.jsx, adminchat.jsx, analytics.jsx ÔÇö Admin views.
7. Inline `<script type="text/babel">` in index.html ÔÇö Login screen + root App component.

There are no imports between files. All components are globals on the window scope via Babel.
- `supabase/functions/` ÔÇö source mirror of the deployed Edge Functions (not served to the browser; for version control / redeploys).
- `adminchat.jsx` is a thin alias (`AdminChatScreen = AdminChat`). The admin "Chat" page chains `admin-messaging` Ôćĺ `BulkMessaging` Ôćĺ `AdminChatScreen` Ôćĺ `AdminChat`, which now simply renders the member `ChatScreen`. The separate admin Channels page (`admin-chat`) was removed.
- Cross-file helpers live on `window`: `fetchSessions/fetchCalSessions/fetchListSessions/mapToCalSession/mapToListSession/AddToCalendar/fmtH` (calendar.jsx); `fetchFocusSessions/focusYmd/focusWeekStart/fmtMins/focusTimerRemaining/useNowTick` (dashboard.jsx); `mapTaskRow` (tasks.jsx); `getActiveUserId`/`_viewingAsId` (index.html ÔÇö see View As below).

### View As (admin impersonation, read-only)
- Triggered from the Members table OR the member detail page (`AdminMemberDetail`). Sets `viewingAs = { id, name, initials, color, plan }`; `isRealAdmin` stays true so the real admin's capabilities are preserved.
- A persistent banner (`RemoteBanner`, rendered inside `.main-inner` above the Topbar so it appears on every route) reads exactly **"You are viewing as [Name]"** with a clearly visible **Exit** button (returns to `admin-members`).
- **Data plumbing:** App holds `viewProfile` (the impersonated member's full profile). When `viewingAs` toggles, App fetches that profile and calls `loadUserData(viewingAs.id)` which refetches tasks/goals/upcoming sessions for the member. The `member` object (used by every screen) derives from `viewProfile || liveProfile` for the visible fields (plan, xp, streak, role, interests, etc.) while `member.isAdmin` stays bound to `liveProfile.role==='admin'` (real capability).
- **Effective user id** is mirrored on `window._viewingAsId` and exposed via `window.getActiveUserId()` (returns `_viewingAsId` if set, else `auth.getUser().id`). Child fetchers that scope reads by user id use this helper: `fetchFocusSessions` (dashboard), `DashStats` attendance, `ProfileScreen` badge effect, `RoadmapScreen`, `fetchResourceViewCount`, `fetchCommunityPostCount`, and `calendar.fetchSessions` (which now applies `attendee_id.eq.<uid>,attendee_id.is.null` explicitly ÔÇö needed because admin's `is_admin()` SELECT override would otherwise return ALL sessions).
- `.main-inner` is wrapped in a `React.Fragment` with `key={effectiveUserId}` so every screen unmounts/remounts on View As toggle ÔÇö this forces all `useEffect([])` data loaders (RoadmapScreen, ProfileScreen, etc.) to re-run with the impersonated id without per-file dep changes.
- **Writes remain on the real admin's auth.uid()** by design (read-only): if the admin clicks a mutator in View As mode, the write targets the member's row and RLS rejects it. Buttons are not visually disabled (acceptable trade-off; if needed, gate UI mutators on `viewingAs`).

## Supabase Integration
URL and anon key are in supabase.jsx. Client is `window._supabase`.

Tables wired to live data:
- `profiles` ÔÇö member auth, role (admin/member), membership_tier ('breakthrough'/'foundations'/'management'), account_type, trial status, member_status, joined_at, `google_connected`, `google_email`. Personal-info cols (editable from Profile, drive the Profile Complete badge): `job_title`, `employment_status`, `field`, `focus_area`, `interests` (text[]). `notification_preferences` (jsonb, with defaults) backs the Profile Ôćĺ Settings notification toggles. XP cols: `xp_total` (int, default 0 ÔÇö cumulative XP, additive only; tier is DERIVED from it, never stored), `streak_days` + `last_streak_date` (weekday-streak engine state, also powers the dashboard Streak widget). **`onboarded_at` (timestamptz, nullable)** ÔÇö server-side first-login gate: App shows the onboarding wizard only when this is null; completion writes `now()` to this column. Existing accounts were backfilled to `joined_at` so they don't re-trigger onboarding. **`legal_accepted_at` (timestamptz, nullable)** ÔÇö server-side legal consent gate: App renders `LegalConsentModal` (modals.jsx) blocking the onboarding wizard, reflection modal, intention modal, and tour bubble whenever this is null. Accept writes `now()` to the column. The column was added with no backfill so EVERY existing account is NULL and will be prompted once on next login. The modal links to `/legal` (Privacy Policy + Terms of Use). (Note: the brief's `xp_regeneration_count` was intentionally SKIPPED ÔÇö regeneration is already tracked per-task in `tasks.regen_count`.)
- `tasks` ÔÇö member tasks. `roadmap_step_id` is now NULLABLE (standalone tasks allowed). Added columns: `goal_id` (direct goal link), `subject`, `priority`, `regen_count` (AI subtask regenerations used, max 3). `points`/`impact` are still client-side defaults (no columns) and `impact` is no longer rendered anywhere; completion uses the real `is_completed` column. Editable from the Tasks detail panel via `EditTaskModal` (title/notes/subject/priority/due_date/goal_id).
- `subtasks` ÔÇö smallest unit of work, lives exclusively under a task. Cols: `task_id` (FK Ôćĺ tasks, ON DELETE CASCADE), `user_id`, `text`, `is_completed`, `source` (`ai`/`manual`), `order_index`, `created_at`. Own-row RLS (select/insert/update/delete) + `is_admin()` override on select/delete. Loaded lazily per task in `TaskDetailPanel`; AI ones come from the `generate-subtasks` Edge Function, manual ones from the inline add input. AI tag = `--sapphire`, Manual tag = `--coral`.
- `goals` ÔÇö member goals (title, description, target_date, status).
- `roadmap_steps` ÔÇö middle layer between goals and tasks (legacy GoalCard link path via `goal_id` OR `roadmap_step.goal_id`; also the phase layer for the Roadmap feature ÔÇö `phase_label`, `reflection_prompt`, `admin_notes`, `order_index`, `status`, week range stored as prefix of `description`).
- `sessions` ÔÇö 1:1s + town halls. Admin creates; members read own 1:1s (`attendee_id = auth.uid()`) + all town halls (`attendee_id` null) via RLS; admins have `is_admin()` SELECT override (View As filters explicitly by attendee_id). Cols: type, title, session_date, start_time, end_time, mentor_name, meeting_link, recurrence, attendee_id, created_by, `google_event_id`, **`agenda`** (text, nullable ÔÇö plain-text bullets, one per line; appended to the Google Calendar invite description by `google-calendar-event` and to .ics/Google/Outlook exports via `sessionDescription()`).
- `wins` ÔÇö community Wins board (load + post). Author info denormalized onto rows (`author_name`, `author_role`, `subject`) since profiles RLS is own-row-only. `is_public` gates the feed.
- `focus_sessions` ÔÇö focus-timer "Save" writes here from BOTH the dashboard pomodoro and the Tasks-page panel timer (duration_minutes = actual elapsed, `selected_minutes` = chosen preset, label, subject, linked_id, linked_kind, started_at). For task-panel sessions `linked_kind='task'` / `linked_id=task.id`; the panel's cumulative "X h Ym focused on this task" line sums duration_minutes for that id. Read by the Focus metrics tab + dashboard "Focus this week" widgets.
- `win_reactions` ÔÇö persisted reactions on wins (win_id, user_id, emoji; unique per trio). SELECT open to authenticated (counts + Win Of The Week computable); insert/delete own. Powers Wins reactions + the Win Of The Week badge.
- `resources` ÔÇö the Resources Library content. Cols: id, title, description, `content_type` (`video`/`pdf`/`template`/`article`), `folder` (`foundations`/`breakthrough`/`mega_management`), subject_area, `access_tier` (same enum as folder), url, thumbnail_url, duration_minutes (videos only), published (default true), created_at. RLS: `resources_admin_all` (admins read/write everything via `is_admin()`) + `resources_member_read` (members SELECT where `published=true` AND `access_tier in ('foundations','breakthrough')` OR `access_tier='mega_management'` AND caller's `membership_tier='management'`). Breakthrough-tier rows ARE returned to Foundations members ÔÇö the lock is a client-side UX gate per the brief (URL is technically exposed to the API even when the card is shown locked). Powers admin Resource Manager + member Library. The legacy `RESOURCES` mock in data.jsx is unused.
- `resource_views` ÔÇö one row per distinct resource a user opened (user_id, resource_id text; unique). Own-row RLS. Powers the Resources badge category. Logged via `window.logResourceView`; counted via `window.fetchResourceViewCount`.
- `channels` ÔÇö chat channels (`name`, `description`, `created_by`, `is_default`, `is_dm`, `created_at`). One seeded `is_default` channel ("MEGA Mentorship") everyone shares. RLS: SELECT = `is_channel_member(id) or is_admin()`; INSERT/UPDATE/DELETE = `is_admin()` (regular channel creation is admin-only). **`is_dm` (boolean, default false)** marks 2-person adminÔćömember direct-message channels ÔÇö these are created via the `get_or_create_dm()` SECURITY DEFINER RPC (not the admin INSERT policy), so any authenticated member can start a DM with the admin.
- `channel_members` ÔÇö channel membership (`channel_id` FK CASCADE, `user_id`, `role`, unique per pair). RLS: co-members + admins can read; admins/owners/self manage rows. The default channel needs NO membership rows (everyone is implicitly a member via `is_default`).
- `messages` ÔÇö chat messages (`channel_id` FK CASCADE, `user_id`, `body`, denormalized `author_name`/`author_role`, `created_at`). RLS: read/write requires `is_channel_member(channel_id) or is_admin()`; author/admin can delete. Powers the chat feed AND the Community badge counter.
- `community_posts` ÔÇö LEGACY single-channel feed (0 rows, superseded by `messages`). Kept but unused; do not build on it.
- `google_tokens` ÔÇö admin Google OAuth tokens. SERVICE-ROLE ONLY (RLS on, no policies); never read client-side.
- `points_log` ÔÇö NOW WIRED as the XP transaction log (reused per the XP brief's "map to existing, don't duplicate" rule instead of a new `xp_transactions` table). One row per XP award: `user_id`, `points` (= xp_awarded), `source` (= action_type), `reason`, `reference_id` (task/subtask/goal/session id), `created_at`. Its `points_log_source_check` was widened to the XP action types (`subtask_complete`/`task_complete`/`goal_complete`/`focus_session`/`town_hall`/`one_to_one`/`streak_bonus`/`badge_unlock`); legacy values kept for back-compat. Town-hall/1:1 attendance counts on the dashboard are derived from this table (own-row SELECT RLS). Written ONLY via the SECURITY DEFINER XP functions.
- `badge_unlocks` ÔÇö one row per badge a member has unlocked (`user_id`, `badge_key` = the badge's display name, unique per pair). Own-row SELECT RLS; written only via `sync_badges`. Gives the +25-per-badge award idempotency.
- `radar_companies` ÔÇö Corporate Radar company database. Cols: id (uuid PK), name, abbreviation, `sector_tags` (text[]), `size_tier` (CHECK: '1-50'/'51-200'/'201-1000'/'1000+'), description, website_url, careers_url, `hiring_detected` (bool), `hiring_last_checked` (timestamptz), logo_color, parent_company, created_at. RLS: authenticated SELECT (all rows readable by members); insert/update/delete service-role only. Seeded with 10 UAE companies.
- `radar_saved_companies` ÔÇö member-saved companies for Corporate Radar. Cols: id (uuid PK), `member_id` (FK profiles, CASCADE), `company_id` (FK radar_companies, CASCADE), `notify` (bool, default false), saved_at. UNIQUE(member_id, company_id). RLS: own-row select/insert/update/delete. The `notify` toggle is gated client-side to paid tiers only.

DB function: `member_count()` ÔÇö SECURITY DEFINER, returns total profile count (members can't read other profiles under RLS, so this powers community/chat member counts).
RLS helper: `is_admin()` ÔÇö SECURITY DEFINER, true when the caller's profile role = 'admin'; used as the admin override across own-row policies.
Chat RLS helpers (all SECURITY DEFINER, anon EXECUTE revoked, authenticated-only): `is_channel_member(cid)` (membership row exists for caller OR channel is_default), `is_channel_owner(cid)` (caller = channels.created_by), `list_members()` (returns id/full_name/role/membership_tier/`xp_total` ÔÇö the member directory for the create-channel multi-select AND the source of other members' XP for the community channel display, since `profiles` is own-row-only under RLS).
XP functions (all SECURITY DEFINER, anon EXECUTE revoked, authenticated-only; XP is additive ÔÇö no deduction logic anywhere):
- `award_xp(p_action, p_xp, p_ref, p_once)` ÔÇö atomic: inserts the `points_log` row AND increments `profiles.xp_total` in one transaction; returns `{awarded, total}`. `p_once=true` skips (returns awarded 0) if a row with the same `(user_id, source, reference_id)` already exists ÔÇö this is how task/subtask/goal/attendance awards stay one-time even if a task is unchecked and re-checked.
- `record_activity(p_today date)` ÔÇö weekday-streak engine. Weekends are ignored (neither advance nor break). Continues the streak when the previous weekday was active (Mon links back to Fri), awarding a +10 `streak_bonus`; otherwise resets to 1. Updates `streak_days`/`last_streak_date`; returns `{bonus, streak, total}`. Client passes its LOCAL date.
- `sync_badges(p_keys text[])` ÔÇö given the currently-earned badge names, inserts any missing `badge_unlocks` rows and awards +25 each (one `points_log` row per badge); idempotent; returns `{awarded, count, total}`.

## Key Patterns and Conventions
- State lives in the root App component in index.html; passed down as props.
- Route persistence: `localStorage.getItem('vantage-route')`.
- **Onboarding gate is server-side** via `profiles.onboarded_at` ÔÇö App sets `showOnboarding=true` only when the loaded profile has `onboarded_at IS NULL`. Tour/intention/reflection state still uses localStorage (per-browser is fine for those).
- **Legal consent gate is server-side** via `profiles.legal_accepted_at` ÔÇö App sets `showLegalConsent=true` whenever it's null and renders `LegalConsentModal` ahead of (and blocking) every other modal/screen. Accept writes `now()`. Same pattern as the onboarding gate; the two run in sequence (legal first, then onboarding).
- Tasks and goals are lifted to App state and passed to both DashboardScreen and TasksScreen.
- Global rewards animation uses `rewards` state array in App (point-pop XP display).
- `SidebarWithWins` wraps `Sidebar` from shell.jsx ÔÇö use this wrapper, not Sidebar directly.
- Do not use ES module imports. All shared functions must be globally accessible.
- Class names in styles.css are not namespaced ÔÇö check for collisions before adding new ones.
- **View As effective user id**: any child screen that needs to scope reads to the current user must use `await window.getActiveUserId()` (returns `_viewingAsId` if set, else the auth user id), NOT `auth.getUser().user.id`. The latter always returns the real admin during View As. Writes should keep using `auth.getUser()` ÔÇö they intentionally fail under RLS when impersonating (read-only enforcement).

### Shared Focus Timer
- There is ONE active focus timer for the whole member app. Its state (`focusTimer`) + the `focus` API (`start/pause/resume/save/reset`) + a `focusTick` counter live in the root App (index.html) and are passed to BOTH `DashboardScreen` and `TasksScreen` as `focus` / `focusTick`. The dashboard pomodoro and the Tasks-panel timer are just views over it, so a session is mirrored across pages and survives collapsing a row / navigating between pages. Starting a new timer replaces any running one (single active session).
- Wall-clock based: the timer object stores `endsAt` (ms) when running and a frozen `remainingSec` when paused; `window.focusTimerRemaining(t)` derives the live remaining (accurate across background-tab throttling). It does NOT survive a full page refresh (no localStorage persistence ÔÇö that was deliberately out of scope).
- Per-second re-renders are localised via `window.useNowTick(active)` (a 1s force-update used only by the components actually showing a countdown), so App does not re-render every second. Collapsed task rows show a live `ÔĆ▒ MM:SS` pill when their timer is running.
- `focus.save()` writes one `focus_sessions` row (elapsed minutes + selected_minutes + linked task) and bumps `focusTick`; `DashboardScreen` reloads its focus rows on `focusTick` and `TaskDetailPanel` re-fetches its cumulative total on `focusTick`.
