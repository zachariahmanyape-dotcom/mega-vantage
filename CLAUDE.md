### View As (admin impersonation, read-only)
- Triggered from the Members table OR the member detail page (`AdminMemberDetail`). Sets `viewingAs = { id, name, initials, color, plan }`; `isRealAdmin` stays true so the real admin's capabilities are preserved.
- A persistent banner (`RemoteBanner`, rendered inside `.main-inner` above the Topbar so it appears on every route) reads exactly **"You are viewing as [Name]"** with a clearly visible **Exit** button (returns to `admin-members`).
- **Data plumbing:** App holds `viewProfile` (the impersonated member's full profile). When `viewingAs` toggles, App fetches that profile and calls `loadUserData(viewingAs.id)` which refetches tasks/goals/upcoming sessions for the member. The `member` object (used by every screen) derives from `viewProfile || liveProfile` for the visible fields (plan, xp, streak, role, interests, etc.) while `member.isAdmin` stays bound to `liveProfile.role==='admin'` (real capability).
- **Effective user id** is mirrored on `window._viewingAsId` and exposed via `window.getActiveUserId()` (returns `_viewingAsId` if set, else `auth.getUser().id`). Child fetchers that scope reads by user id use this helper: `fetchFocusSessions` (dashboard), `DashStats` attendance, `ProfileScreen` badge effect, `RoadmapScreen`, `fetchResourceViewCount`, `fetchCommunityPostCount`, and `calendar.fetchSessions` (which now applies `attendee_id.eq.<uid>,attendee_id.is.null` explicitly â€” needed because admin's `is_admin()` SELECT override would otherwise return ALL sessions).
- `.main-inner` is wrapped in a `React.Fragment` with `key={effectiveUserId}` so every screen unmounts/remounts on View As toggle â€” this forces all `useEffect([])` data loaders (RoadmapScreen, ProfileScreen, etc.) to re-run with the impersonated id without per-file dep changes.
- **Writes remain on the real admin's auth.uid()** by design (read-only): if the admin clicks a mutator in View As mode, the write targets the member's row and RLS rejects it. Buttons are not visually disabled (acceptable trade-off; if needed, gate UI mutators on `viewingAs`).

DB function: `member_count()` â€” SECURITY DEFINER, returns total profile count (members can't read other profiles under RLS, so this powers community/chat member counts).

XP functions (all SECURITY DEFINER, anon EXECUTE revoked, authenticated-only; XP is additive â€” no deduction logic anywhere):
- `award_xp(p_action, p_xp, p_ref, p_once)` â€” atomic: inserts the `points_log` row AND increments `profiles.xp_total` in one transaction; returns `{awarded, total}`. `p_once=true` skips (returns awarded 0) if a row with the same `(user_id, source, reference_id)` already exists â€” this is how task/subtask/goal/attendance awards stay one-time even if a task is unchecked and re-checked.
- `record_activity(p_today date)` â€” weekday-streak engine. Weekends are ignored (neither advance nor break). Continues the streak when the previous weekday was active (Mon links back to Fri), awarding a +10 `streak_bonus`; otherwise resets to 1. ### Shared Focus Timer
- There is ONE active focus timer for the whole member app. Its state (`focusTimer`) + the `focus` API (`start/pause/resume/save/reset`) + a `focusTick` counter live in the root App (index.html) and are passed to BOTH `DashboardScreen` and `TasksScreen` as `focus` / `focusTick`. The dashboard pomodoro and the Tasks-panel timer are just views over it, so a session is mirrored across pages and survives collapsing a row / navigating between pages. Starting a new timer replaces any running one (single active session).
- Wall-clock based: the timer object stores `endsAt` (ms) when running and a frozen `remainingSec` when paused; `window.focusTimerRemaining(t)` derives the live remaining (accurate across background-tab throttling). It does NOT survive a full page refresh (no localStorage persistence â€” that was deliberately out of scope).
- Per-second re-renders are localised via `window.useNowTick(active)` (a 1s force-update used only by the components actually showing a countdown), so App does not re-render every second. Collapsed task rows show a live `âŹ± MM:SS` pill when their timer is running.
- `focus.save()` writes one `focus_sessions` row (elapsed minutes + selected_minutes + linked task) and bumps `focusTick`; `DashboardScreen` reloads its focus rows on `focusTick` and `TaskDetailPanel` re-fetches its cumulative total on `focusTick`.
